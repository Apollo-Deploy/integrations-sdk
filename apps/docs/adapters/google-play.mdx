## Overview

The Google Play Console adapter provides comprehensive access to the **Google Play Developer API** for Android apps. It uses OAuth 2.0 with long-lived refresh tokens for user authorization, and the **Edits API** pattern — all content changes are staged in a draft "edit" and committed atomically. Real-time notifications are delivered via **Google Cloud Pub/Sub**.

**Capability:** [`app-store`](/docs/capabilities/app-store)

**Auth model:** OAuth 2.0 + refresh tokens

**Token lifetime:** 1 hour, refreshable

---

## Prerequisites

1. Go to [Google Cloud Console](https://console.cloud.google.com) → **APIs & Services → Credentials → Create Credentials → OAuth 2.0 Client ID**
2. Select **Web application**, add an **Authorized redirect URI** (e.g. `https://yourapp.com/oauth/callback/google-play`)
3. Copy **Client ID** and **Client Secret**
4. Enable the **Google Play Android Developer API** in the Cloud Console
5. In [Google Play Console](https://play.google.com/console): **Setup → API access → Link to Google Cloud project**, then grant your OAuth app access to the Play Console account
6. For real-time notifications: Set up a **Cloud Pub/Sub** topic and push subscription at `https://yourapp.com/webhooks/google-play`

## Registering with the hub

```typescript title="lib/integrations.ts" lineNumbers
import { IntegrationHub } from '@apollo-deploy/integrations';
import { createGooglePlayAdapter } from '@apollo-deploy/adapter-google-play';

const hub = new IntegrationHub();

hub.register('google-play', createGooglePlayAdapter({
  clientId: process.env.GOOGLE_CLIENT_ID!,
  clientSecret: process.env.GOOGLE_CLIENT_SECRET!,
  webhookSecret: process.env.GOOGLE_PLAY_WEBHOOK_SECRET,
}));

await hub.initialize();

hub.onEvent('google-play', (event) => {
  if (event.eventType === 'subscription.renewed') {
    const data = event.data as any;
    console.log('Subscription renewed:', data.subscriptionNotification?.purchaseToken);
  }
});
```

## Architecture

The Google Play adapter's `app-store` capability is split into **domain-specific modules** composed at initialization. Each module is a factory that returns a `Pick<AppStoreCapability, ...>` partial:

```
adapter-google-play/src/capabilities/
├── _context.ts          # GooglePlayContext, Edits API helpers, withEdit()
├── apps.ts              # listApps, getApp
├── builds.ts            # listBuilds, getBuild, artifacts, uploadBinary
├── releases.ts          # releases, tracks, versions, publishing
├── reviews.ts           # reviews, ratings
├── beta.ts              # beta groups (internal/alpha/beta tracks), testers
├── vitals.ts            # Android Vitals, crash clusters, ANR clusters
├── recovery.ts          # App Recovery actions (Google Play only)
├── phased-releases.ts   # staged rollout lifecycle
└── index.ts             # composes all modules into AppStoreCapability
```

The barrel `index.ts` creates a shared `GooglePlayContext` (OAuth-authenticated fetch, Edits API `withEdit()` helper, base URLs) and spreads each module's methods into a single `AppStoreCapability`:

```typescript
export function createGooglePlayAppStore(config: GooglePlayAdapterConfig): AppStoreCapability {
  const ctx = createGooglePlayContext(config);

  return {
    ...createGooglePlayApps(ctx),
    ...createGooglePlayBuilds(ctx),
    ...createGooglePlayReleases(ctx),
    ...createGooglePlayReviews(ctx),
    ...createGooglePlayBeta(ctx),
    ...createGooglePlayVitals(ctx),
    ...createGooglePlayRecovery(ctx),
    ...createGooglePlayPhasedReleases(ctx),
  };
}
```

## OAuth flow

### Step 1 — Generate the authorization URL

```typescript title="lib/google-play-oauth.ts" lineNumbers
const adapter = hub.getAdapter('google-play');

const url = adapter.oauth.getAuthorizationUrl({
  state: crypto.randomUUID(),  // Store in session for CSRF
  scopes: ['https://www.googleapis.com/auth/androidpublisher'],
  redirectUri: 'https://yourapp.com/oauth/callback/google-play',
  extras: {
    access_type: 'offline',  // REQUIRED to get a refresh token
    prompt: 'consent',       // REQUIRED to get refresh token on re-auth
  },
});
```

<Callout type="warn">
Always pass `access_type: 'offline'` and `prompt: 'consent'` in `extras`. Without these, Google will not return a refresh token — or will only return one on the **first** authorization. Re-authorization without `prompt: 'consent'` will return an access token only.
</Callout>

### Available scopes

| Scope | Description |
|-------|-------------|
| `https://www.googleapis.com/auth/androidpublisher` | Full access to Google Play Developer API |

### Step 2 — Exchange the code

```typescript title="app/oauth/callback/google-play/route.ts" lineNumbers
const tokens = await adapter.oauth.exchangeCode({
  code: searchParams.get('code')!,
  redirectUri: 'https://yourapp.com/oauth/callback/google-play',
});

// tokens.accessToken    — Bearer token (1 hour)
// tokens.refreshToken   — Long-lived refresh token
// tokens.expiresAt      — 1 hour from now
// tokens.providerData   — see below
```

### `tokens.providerData` reference

| Key | Type | Description |
|-----|------|-------------|
| `email` | `string` | Google account email of the authorizing user |

### Step 3 — Refresh tokens

Google refresh tokens are long-lived (they only expire if revoked or unused for 6 months).

```typescript
const refreshed = await adapter.oauth.refreshToken(tokens.refreshToken!);
// refreshed.accessToken — new 1-hour token
// refreshed.refreshToken — same refresh token (non-rotating)
// refreshed.expiresAt   — 1 hour from now
```

## Real-time notifications (Pub/Sub)

Google Play delivers developer notifications via **Cloud Pub/Sub push subscriptions**. These are base64-encoded JSON messages, not standard HTTP webhook payloads.

### Setting up Pub/Sub notifications

1. In Google Play Console: **Monetize → Monetization Setup → Real-time developer notifications**
2. Enter your **Pub/Sub topic** (e.g. `projects/my-project/topics/play-notifications`)
3. In Cloud Console: Create a **push subscription** on the topic, targeting `https://yourapp.com/webhooks/google-play`

```typescript title="app/webhooks/google-play/route.ts" lineNumbers
import { NextRequest } from 'next/server';

export async function POST(req: NextRequest) {
  const rawBody = Buffer.from(await req.arrayBuffer());
  const headers = Object.fromEntries(req.headers.entries());

  const result = await hub.webhooks['google-play']({ rawBody, headers });
  return Response.json({ ok: true }, { status: result.statusCode ?? 200 });
}
```

### Notification event types

| Pub/Sub `notificationType` | `subscriptionNotification?` | Normalized `eventType` |
|---|---|---|
| SUBSCRIPTION_PURCHASED | `1` | `subscription.purchased` |
| SUBSCRIPTION_RENEWED | `2` | `subscription.renewed` |
| SUBSCRIPTION_CANCELLED | `3` | `subscription.cancelled` |
| SUBSCRIPTION_PURCHASED_FROM_FREE_TRIAL | `12` | `subscription.trial_converted` |
| SUBSCRIPTION_EXPIRED | `13` | `subscription.expired` |
| ONE_TIME_PRODUCT_PURCHASED | `-` | `purchase.completed` |
| ONE_TIME_PRODUCT_CANCELED | `-` | `purchase.cancelled` |

## App Store capability

<Callout type="info">
The Google Play adapter uses the **Edits API** pattern. All write operations (updating tracks, releases) open a draft edit, modify it, then commit. The adapter handles this automatically via the `withEdit()` helper in `_context.ts`.
</Callout>

```typescript
const store = hub.getAdapter('google-play').appStore!;
```

### App management

<small>`capabilities/apps.ts`</small>

#### `getApp(tokens, packageName)`

Fetch app information. Use the **package name** (e.g. `com.example.myapp`) as the app ID.

```typescript
const app = await store.getApp(tokens, 'com.example.myapp');
// { id: 'com.example.myapp', name: 'My App', platform: 'android', ... }
```

<Callout type="warn">
`listApps` is **not supported** by the Google Play Developer API. Google Play API does not provide an endpoint to list all apps. You must store package names in your own database and pass them directly to capability methods.
</Callout>


#### `listBuildArtifacts(tokens, packageName, buildId)`

List AAB/APK artifacts for a specific versionCode.

```typescript
const artifacts = await store.listBuildArtifacts(tokens, 'com.example.myapp', '42');
// [{ id, buildId, type: 'aab' | 'apk', fileName, ... }]
```

#### `getArtifactDownloadUrl(tokens, packageName, buildId, artifactId)`

```typescript
const { url, expiresAt } = await store.getArtifactDownloadUrl(
  tokens, 'com.example.myapp', '42', 'artifact-id'

```

#### `uploadBinary(tokens, packageName, file, opts?)`

Upload an AAB or APK to the Play Store. Supports two channels:

```typescript
// Store upload (default) — stages in an Edit, optionally commits
const result = await store.uploadBinary(tokens, 'com.example.myapp', aabFile, {
  fileType: 'aab',
  channel: 'store',           // default
  commitEdit: true,           // default — commit the Edit after upload
  deviceTierConfigId: 'LATEST',
});
// result: { channel: 'store', fileType: 'aab', build: StoreBuild }

// Internal App Sharing — instant download URL for testers
const result = await store.uploadBinary(tokens, 'com.example.myapp', apkFile, {
  fileType: 'apk',
  channel: 'internal-sharing',
});
// result: { channel: 'internal-sharing', fileType: 'apk', artifact: InternalSharingArtifact }
// artifact.downloadUrl — shareable install link
```

**Upload options:**

| Option | Type | Default | Description |
|--------|------|---------|-------------|
| `fileType` | `'aab' \| 'apk'` | Auto-detected | Explicit override for file type |
| `channel` | `'store' \| 'internal-sharing'` | `'store'` | Distribution channel |
| `commitEdit` | `boolean` | `true` | Whether to commit the Edit after upload (`store` only) |
| `deviceTierConfigId` | `string?` | — | DTC ID for APK generation from AAB (`store` only) |

### Track management

<small>`capabilities/releases.ts`</small>

#### `listTracks(tokens, packageName)`

```typescript
const tracks = await store.listTracks(tokens, 'com.example.myapp');

//   { id: 'com.example.myapp:production', name: 'production', type: 'production', releases: [...] },
```

#### `getTrack(tokens, packageName, trackId)`

`trackId` can be the full `packageName:trackName` or just the track name.

```typescript

```

#### `updateTrack(tokens, packageName, trackId, update)`

Modify rollout state for releases on a track. **Commits the edit atomically.**

```typescript
// Increase rollout to 50%
await store.updateTrack(tokens, 'com.example.myapp', 'production', {
  rolloutPercentage: 50,
});

// Halt a rollout
await store.updateTrack(tokens, 'com.example.myapp', 'production', { halt: true });

// Resume a halted rollout
await store.updateTrack(tokens, 'com.example.myapp', 'production', { resume: true });

// Complete rollout (100%)
await store.updateTrack(tokens, 'com.example.myapp', 'production', { completeRollout: true });
```

#### `listVersions(tokens, packageName)` / `getVersion(tokens, packageName, versionId)`

```typescript

const version = await store.getVersion(tokens, 'com.example.myapp', 'version-id');
```

### Phased releases (Staged rollouts)

<small>`capabilities/phased-releases.ts`</small>

Google Play uses manually-controlled **staged rollouts** with a `userFraction` percentage.

#### `createPhasedRelease(tokens, packageName, request)`

Start a staged rollout for a track release. The `versionId` must be in `track:versionCode` format.

```typescript
const phased = await store.createPhasedRelease(tokens, 'com.example.myapp', {
  versionId: 'production:42',
  rolloutPercentage: 1,  // Start at 1%
});
// { id, versionId, state: 'active', rolloutPercentage: 1 }
```

#### `getPhasedRelease(tokens, packageName, versionId)`

```typescript
const phased = await store.getPhasedRelease(tokens, 'com.example.myapp', 'production:42');
// { id, versionId, state: 'active', rolloutPercentage: 20 }
```

#### `updatePhasedRelease(tokens, packageName, phasedReleaseId, update)`

Increase rollout, pause, resume, or complete.

```typescript
// Increase to 50%
await store.updatePhasedRelease(tokens, 'com.example.myapp', 'phased-id', {
  rolloutPercentage: 50,
});

// Pause the rollout
await store.updatePhasedRelease(tokens, 'com.example.myapp', 'phased-id', {
  state: 'paused',
});

// Resume
await store.updatePhasedRelease(tokens, 'com.example.myapp', 'phased-id', {
  state: 'active',
});

// Complete — release to 100%
await store.updatePhasedRelease(tokens, 'com.example.myapp', 'phased-id', {
  state: 'complete',
});
```

#### `deletePhasedRelease(tokens, packageName, phasedReleaseId)`

Halt the staged rollout on the track.

```typescript
await store.deletePhasedRelease(tokens, 'com.example.myapp', 'phased-id');
```

#### `createReleaseRequest(tokens, packageName, versionId)`

Manually release a version to production (commits the track with `completed` status).

```typescript
const request = await store.createReleaseRequest(tokens, 'com.example.myapp', 'production:42');
// { id, versionId, requestedAt: Date }
```

### App Recovery

<small>`capabilities/recovery.ts`</small>

App Recovery lets you remotely remediate issues for users on affected app versions. This is a **Google Play–only feature** — the Apple adapter throws `CapabilityError` for these methods.

#### `listAppRecoveryActions(tokens, packageName, opts?)`

```typescript
const { items } = await store.listAppRecoveryActions(tokens, 'com.example.myapp', {
  versionCode: '42',
  limit: 10,
});
// [{ appRecoveryId, appId, status: 'ACTIVE_RECOVERING', targeting, createTime }]
```

**Recovery status values:**

| Status | Description |
|--------|-------------|
| `DRAFT` | Created but not deployed |
| `ACTIVE_RECOVERING` | Actively being applied to users |
| `ACTIVE_WAITING` | Deployed, waiting for conditions |
| `COMPLETE` | Recovery fully applied |
| `CANCELLED` | Recovery cancelled |

#### `createAppRecoveryAction(tokens, packageName, request)`

Create a new recovery action in `DRAFT` status.

```typescript
const action = await store.createAppRecoveryAction(tokens, 'com.example.myapp', {
  versionCodes: ['40', '41'],
  remediationType: 'REMOTE_IN_APP_UPDATE',
});
```

| Remediation type | Description |
|--|--|
| `REMOTE_IN_APP_UPDATE` | Prompt users to update to a fixed version |
| `FORCE_CRASH` | Force-crash the app to trigger an update |

#### `deployAppRecoveryAction(tokens, packageName, recoveryId)`

Deploy a `DRAFT` recovery action so it starts reaching affected users.

```typescript
const deployed = await store.deployAppRecoveryAction(tokens, 'com.example.myapp', 'recovery-id');
```

#### `cancelAppRecoveryAction(tokens, packageName, recoveryId)`

Cancel an active or draft recovery.

```typescript
const cancelled = await store.cancelAppRecoveryAction(tokens, 'com.example.myapp', 'recovery-id');
```

#### `addAppRecoveryTargeting(tokens, packageName, recoveryId, targeting)`

Expand the targeting of an existing recovery action.

```typescript
await store.addAppRecoveryTargeting(tokens, 'com.example.myapp', 'recovery-id', {
  versionCodes: ['42'],
  regions: { includeList: ['US', 'CA'] },
});
```

## Error handling

| Scenario | Error class | Retryable |
|----------|-------------|-----------|
| Expired access token | `TokenRefreshError` | Yes |
| Package not found / access denied | `CapabilityError` | No |
| Rate limited (429) | `CapabilityError` | Yes |
| Edit conflict / stale edit | `CapabilityError` | Yes — retry |
| `listApps` called | `CapabilityError` | No — not supported |
| `deleteReviewReply` called | `CapabilityError` | No — not supported |

## Common gotchas

**Edits API commits:** Write operations (`updateTrack`, `releaseToTrack`, `submitForReview`, `uploadBinary`) use `withEdit(..., { commit: true })` — they open an edit, apply changes, and commit in one atomic operation. Read operations open an edit then immediately delete it (no commit). If a write operation fails mid-flight, the adapter cleans up the draft edit automatically.

**`listApps` not supported:** Google Play API does not expose an apps list endpoint. Store package names in your own database and use them directly (e.g. `com.example.myapp`). The `getApp` method opens an edit to confirm the package is accessible.

**Pub/Sub message format:** Google sends base64-encoded JSON wrapped in a Pub/Sub envelope. The adapter unwraps the envelope and parses the inner `developerNotification` payload automatically.

**Track promotion:** There is no dedicated "promote from beta to production" endpoint. Create a new production release with the same versionCode(s) via `releaseToTrack` with `track: 'production'`.

**Review replies:** You can only reply to reviews that have `replyAllowed: true`. New reviews take a short time to become replyable. Deleting replies is not supported by the API.

**Staged rollout vs phased release:** Google Play allows arbitrary rollout percentages via `updatePhasedRelease` or `updateRollout`. Unlike Apple's fixed 7-day schedule, you control the exact percentage and timing.

**Internal App Sharing:** Use `uploadBinary` with `channel: 'internal-sharing'` to get an instant download URL for internal testers, bypassing the review and track system entirely.

**App Recovery targeting:** When creating a recovery action, you can target by version code list, version range, or all users. You can expand targeting after deployment with `addAppRecoveryTargeting`, but you cannot narrow it.
