## Overview

The Apple App Store Connect adapter provides comprehensive access to the **App Store Connect API** for iOS, macOS, tvOS, and watchOS apps. It authenticates using a **short-lived JWT** signed with your private key (ES256) — no OAuth flow, no user authorization. Access is granted via API keys created in App Store Connect.

**Capability:** [`app-store`](/docs/capabilities/app-store)

**Auth model:** JWT (API key-based) — no user OAuth

**Token lifetime:** 20 minutes maximum per JWT (fresh token generated per request)

---

## Prerequisites

1. Go to [appstoreconnect.apple.com](https://appstoreconnect.apple.com) → **Users and Access → Keys**
2. Click **"+"** to generate a new API key
3. Choose a **Role** (Admin for full access, or a more restricted role)
4. Download the **`.p8` private key** file — you can only download it **once**
5. Note the **Key ID** and **Issuer ID** from the keys page
6. For the vendor number (financial reports): go to **Payments and Financial Reports → your vendor number**
7. For App Store Server Notifications: set your URL in your app's **App Store → App Information → App Store Server Notifications**

## Architecture

The Apple adapter's `app-store` capability is split into **domain-specific modules** composed at initialization. Each module is a factory that returns a `Pick<AppStoreCapability, ...>` partial:

```
adapter-apple/src/capabilities/
├── _context.ts          # AppleContext, JWT auth, API helpers
├── apps.ts              # listApps, getApp
├── builds.ts            # listBuilds, getBuild, artifacts, uploadBinary
├── releases.ts          # releases, tracks, versions, publishing
├── reviews.ts           # reviews, ratings
├── beta.ts              # TestFlight groups, testers, build assignment
├── vitals.ts            # crash clusters, ANR, performance metrics
├── recovery.ts          # App Recovery (throws — Google Play only)
├── phased-releases.ts   # phased release lifecycle
└── index.ts             # composes all modules into AppStoreCapability
```

The barrel `index.ts` creates a shared `AppleContext` (JWT generation, authenticated fetch helper, cursor extraction) and spreads each module's methods into a single `AppStoreCapability`:

```typescript
export function createAppleAppStore(config: AppleAdapterConfig): AppStoreCapability {
  const ctx = createAppleContext(config);

  return {
    ...createAppleApps(ctx),
    ...createAppleBuilds(ctx),
    ...createAppleReleases(ctx),
    ...createAppleReviews(ctx),
    ...createAppleBeta(ctx),
    ...createAppleVitals(ctx),
    ...createAppleRecovery(),       // all methods throw CapabilityError
    ...createApplePhasedReleases(ctx),
  };
}
```

## Authentication

App Store Connect uses **JSON Web Tokens (JWT)** signed with ES256 (ECDSA + SHA-256). A new JWT is generated automatically when an existing token is within 60 seconds of expiry.

**JWT claims:**

| Claim | Value |
|-------|-------|
| `iss` | `issuerId` |
| `iat` | Current Unix timestamp |
| `exp` | `iat + 1200` (20 minutes max) |
| `aud` | `appstoreconnect-v1` |

The JWT is sent as `Authorization: Bearer <jwt>` on every request.

<Callout type="info">
There is no OAuth flow for Apple App Store Connect. The `oauth` handler exists for interface compatibility but generates a JWT from your private key rather than redirecting a user. You call `adapter.oauth.exchangeCode({ code: '' })` to get an initial `TokenSet` — store this and use it for all API calls.
</Callout>

## App Store Server Notifications

Apple sends server-to-server notifications for in-app purchase events. These are payload-level `signedPayload` JWTs verified against Apple's root certificates.

```typescript title="app/webhooks/apple/route.ts" lineNumbers
import { NextRequest } from 'next/server';

export async function POST(req: NextRequest) {
  const rawBody = Buffer.from(await req.arrayBuffer());
  const headers = Object.fromEntries(req.headers.entries());

  const result = await hub.webhooks.apple({ rawBody, headers });
  return Response.json({ ok: true }, { status: result.statusCode ?? 200 });
}
```

### Supported notification types

| Notification `notificationType` | Normalized `eventType` | Description |
|---|---|---|
| `DID_RENEW` | `subscription.renewed` | Subscription successfully renewed |
| `SUBSCRIBED` | `subscription.started` | New subscription |
| `DID_CHANGE_RENEWAL_STATUS` | `subscription.renewal_status_changed` | Auto-renewal toggled |
| `DID_FAIL_TO_RENEW` | `subscription.renewal_failed` | Billing failure |
| `EXPIRED` | `subscription.expired` | Subscription fully expired |
| `REFUND` | `subscription.refunded` | Purchase refunded |
| `CONSUMPTION_REQUEST` | `subscription.consumption_request` | Consumable refund prompt |
| `REVOKE` | `entitlement.revoked` | Family Sharing revocation |

### Phased releases

<small>`capabilities/phased-releases.ts`</small>

Apple uses a **fixed 7-day schedule** for phased releases: 1% → 2% → 5% → 10% → 20% → 50% → 100%.

#### `createPhasedRelease(tokens, appId, request)`

Enable phased release for an App Store version. Creates the resource in `inactive` state — the schedule starts automatically when the version goes live.

```typescript
const phased = await store.createPhasedRelease(tokens, '1234567890', {
  versionId: 'version-uuid',
});
// { id, versionId, state: 'inactive', rolloutPercentage: 0 }
```

#### `getPhasedRelease(tokens, appId, versionId)`

```typescript
const phased = await store.getPhasedRelease(tokens, '1234567890', 'version-uuid');
// { id, versionId, state: 'active', currentDayNumber: 3, rolloutPercentage: 10 }
```

#### `updatePhasedRelease(tokens, appId, phasedReleaseId, update)`

Pause, resume, or complete a phased release.

```typescript
// Pause the rollout
await store.updatePhasedRelease(tokens, '1234567890', 'phased-id', { state: 'paused' });

// Resume
await store.updatePhasedRelease(tokens, '1234567890', 'phased-id', { state: 'active' });

// Complete — release to 100% immediately
await store.updatePhasedRelease(tokens, '1234567890', 'phased-id', { state: 'complete' });
```

#### `deletePhasedRelease(tokens, appId, phasedReleaseId)`

Cancel a planned phased release that has not started (state must be `inactive`).

```typescript
await store.deletePhasedRelease(tokens, '1234567890', 'phased-id');
```

#### `createReleaseRequest(tokens, appId, versionId)`

Manually release a version in "Pending Developer Release" state. **Irreversible.**

```typescript
const request = await store.createReleaseRequest(tokens, '1234567890', 'version-uuid');
// { id, versionId, requestedAt: Date }
```

## Common gotchas

**App Store Connect API keys have roles:** Choose the appropriate role. `Customer Support` can respond to reviews. `Developer` can manage builds and testing. `Admin` has full access.

**Phased release requires prior setup:** You can only control a phased release if `appStoreVersionPhasedRelease` was set up during the initial version creation. Use `createPhasedRelease` to set it up before submitting for review.

**JWT expiry limit:** Apple enforces a maximum JWT lifetime of **20 minutes** (`exp - iat ≤ 1200`). The adapter generates fresh JWTs with this maximum.

**`listApps` limitation:** The Apps endpoint returns all apps associated with your API key's team. If your key is scoped to a specific app, `listApps` may return only that app.

**Reviews are per-territory:** `listReviews` returns reviews from all territories by default. Use the `territory` parameter to filter to a specific App Store storefront (e.g. `'GBR'`, `'USA'`, `'JPN'`).

**IPA uploads require the CLI:** There is no REST endpoint for uploading IPA binaries. Use `xcrun altool --upload-app` or the Transporter app. The `uploadBinary` method will throw a `CapabilityError` with the exact command to run.

**App Recovery is unsupported:** App Recovery is a Google Play–only feature. All five recovery methods throw `CapabilityError` on the Apple adapter.

**Phased release day schedule:** Apple's 7-day phased rollout follows a fixed schedule (1% → 2% → 5% → 10% → 20% → 50% → 100%). You cannot set a custom percentage — use `updatePhasedRelease` to pause/resume/complete only.
